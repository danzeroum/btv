O que o repositório já é

Não é um wrapper de bpmn-js: é uma implementação própria (SVG nativo + TypeScript puro) em três camadas — core headless (modelo, comandos com undo, eventos com veto, lifecycle, regras, validação, diff, ledger, geometria, XML BPMN 2.0 com DI), camada React (canvas, formas, gestos, paleta, inspetor, minimap, DiffView) e plugin de domínio de exemplo. O vocabulário do Squad Designer (papéis, gates, conectores, entregas) entra por plugin declarativo — exatamente a arquitetura certa para manter a biblioteca agnóstica.

04

Proposta: 2 módulos novos + melhorias pontuais

Sim, vale um módulo para "tratar versionamento mais fácil" — mas como camada de registro sobre o LifecycleEngine, não substituto. Nomes agnósticos, seguindo o padrão do repo:

@bpmn-react/registry novo · prioridade 1

O elo que falta entre versões soltas e governança consultável. Um VersionRegistry headless (mesmo padrão sink do ledger) que guarda a cadeia de versões + snapshots e responde as perguntas de auditoria da discussão:

  • activeAt(date) — "qual versão estava em vigor no dia X?" (linha do tempo por effectiveFrom/Until)
  • history() / diffBetween(v1, v2) — cadeia completa com snapshot + diff estruturado (snapshot para executar, diff para explicar)
  • Canais e ambientes — publish(version, { channel: 'internal' | 'pilot' | 'general', environment }): uma versão pode estar active no piloto e candidate na geral, cobrindo rollout gradual sem flag solta
  • Changelog dual — changeSummary (negócio) + technicalNotes ligado ao diff (auditoria)
@bpmn-react/run-binding novo · prioridade 2

Pinagem de execução — o "commit hash do deploy" aplicado a processo: bindRun(diagram) devolve um registro imutável { versionId, semanticVersion, snapshotHash, channel } para o host gravar em cada execução/entrega. Regra embutida: execução nasce presa à versão; mudança de versão nunca altera run em andamento. Pequeno o suficiente para talvez viver dentro do registry — decidir na implementação; o contrato é o que importa.

05

Agnosticismo e fronteira biblioteca × produto

O repositório já está limpo: nomes genéricos (bpmn-react, domain-example), vocabulário de domínio isolado em plugin, e o README não menciona o produto de origem. Para manter assim, a fronteira recomendada:

fica na biblioteca
  • Canvas, formas, gestos, paleta, inspetor
  • Lifecycle, registry, canais, run-binding
  • Ledger, diff, XML BPMN 2.0
  • Selo de status + timeline (componentes genéricos)
fica no produto (plugin/host)
  • Vocabulário: papéis, personas, gates, entregas
  • Prompts padrão e restauração ao padrão
  • Execução das squads (esteira ao vivo, cockpit)
  • Autenticação, papéis de aprovador, notificação a cliente
